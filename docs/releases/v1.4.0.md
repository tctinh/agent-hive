# Release v1.4.0

Compare: [`main...feature/context-model-clarification`](https://github.com/tctinh/agent-hive/compare/main...feature/context-model-clarification)

---

## Highlights

### Clarified Context Model and Reserved Context Semantics

`v1.4.0` makes Hive's memory model more explicit. `context/overview.md` is the primary human-facing branch summary/history, `plan.md` remains execution truth, `draft` stays planner-only scratch space, and `execution-decisions` records orchestration decisions without pretending to be user-facing documentation.

**Why it matters:** Hive works best when every artifact has one clear job. This release reduces ambiguity about which files humans review, which files workers execute against, and which context can safely graduate into durable agent memory.

### Deterministic Hive Network Foundation (PR #75)

PR #75 adds the first deterministic Hive Network foundation through `NetworkService` and `hive_network_query`. Planning, orchestration, and review roles can now query prior features for network-safe snippets with deterministic ordering and explicit JSON output, while live-file verification remains mandatory.

**Why it matters:** Historical lookup is useful only when it is narrow, predictable, and subordinate to the live repository. `v1.4.0` adds retrieval without turning Hive into a fuzzy memory system or an embeddings-first RAG product.

### OpenCode-First Support Policy

This release makes the support policy explicit: OpenCode remains the first-class supported execution harness, while `vscode-hive` remains a companion for overview-first review, sidebar visibility, approvals, and task-launch UX around the `.hive/` workspace.

**Why it matters:** Issue #70 made the strategic direction clear. GitHub Copilot support was useful as an experiment, but it is no longer strong enough to be a first-class supported path for Hive's planning and execution workflow.

### Recovery and Helper Isolation Hardening Ship on the Same Branch

The branch prepared for `v1.4.0` also carries the recent hardening work around compaction/recovery boundaries and `hive-helper` isolation. Helper flows remain merge-only and network-blind, and recovery guidance stays more tightly bounded to the active task/runtime role.

**Why it matters:** The new context and network capabilities are only useful if runtime boundaries stay disciplined. `v1.4.0` pairs new retrieval power with stronger isolation and recovery rules.

## What Changed

### New: deterministic historical lookup for planning/orchestration/review
- Added `packages/hive-core/src/services/networkService.ts`
- Added `hive_network_query` to the OpenCode plugin with deterministic JSON results
- Limited network visibility to Hive / Architect / Swarm / Hygienic roles
- Kept Forager, Scout, and `hive-helper` out of the network surface

### Changed: context roles are now more explicit
- Treated `context/overview.md` as the primary human-facing summary/history artifact
- Kept `plan.md` as execution truth for workers
- Reserved `draft` for planning scratch work and `execution-decisions` for orchestration history
- Kept AGENTS graduation conservative by syncing only durable context

### Changed: support story is now honest and narrower
- OpenCode remains the first-class supported harness
- `vscode-hive` remains supported as a review/sidebar companion instead of the primary execution surface
- GitHub Copilot and VS Code language-model-tool integration are no longer first-class supported paths from `v1.4.0` onward
- Existing `.github/*` bootstrap generation is intentionally retained for continuity, but it is not the recommended primary workflow

### Fixed / Hardened: helper isolation and recovery boundaries
- Preserved `hive-helper` as merge-only, not a general execution agent
- Kept helper flows network-blind even as Hive Network was introduced elsewhere
- Tightened compaction/recovery guidance so resumed sessions re-anchor to the correct role and task boundary more reliably

## Breaking Changes

None at the feature-directory or data-model level. Existing `.hive/` nests continue to work without a manual migration.

## Support Policy Notes

- OpenCode is the first-class supported harness for Hive planning and execution.
- `vscode-hive` remains supported for review/sidebar UX around the `.hive/` workspace.
- GitHub Copilot support is no longer first-class from `v1.4.0` onward.
- Retained `.github/*` bootstrap artifacts are continuity helpers, not the recommended long-term path.

---

## Upgrade Notes

- If you still rely on Copilot-driven Hive flows, move day-to-day execution to OpenCode.
- Keep `vscode-hive` installed if you want overview-first review, sidebar visibility, approval UX, and task launching around OpenCode.
- Treat `context/overview.md` as the branch summary/history, keep `plan.md` as execution truth, and use reserved context files only for their intended roles.
- Use Hive Network as optional historical lookup for planning, orchestration, and review, then verify against live files before acting.
