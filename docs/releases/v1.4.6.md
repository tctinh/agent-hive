# Release v1.4.6

Compare: [`v1.4.5...v1.4.6`](https://github.com/tctinh/agent-hive/compare/v1.4.5...v1.4.6)

---

## Highlights

### 1 · MCP runtime package renamed to `@tctinh/agent-hive-mcp`

The MCP runtime package ships under the scoped name `@tctinh/agent-hive-mcp` starting with this release. The previous name `hive-mcp` was taken on npm.

All internal references — `claude-code-hive` dependency declarations, module specifiers in the launch/verify scripts, the publish preflight, and the integration test — have been updated to use the new name. The package directory on disk remains `packages/hive-mcp`; only the npm-facing identity changes.

**Why it matters:** A package that cannot publish to its intended npm name is broken by definition. The rename resolves the blocked publish path before the first public release.

### 2 · The first `claude-code-hive` publish is now part of the normal release path

`v1.4.6` closes the gap between the release workflow and the npm preflight for the first Claude package release.

The tagged release path already built, tested, packed, and ordered `claude-code-hive` behind `@tctinh/agent-hive-mcp`, but the npm preflight still assumed every package already existed and already had collaborator metadata. That assumption was correct for existing packages and wrong for a brand-new one.

This patch teaches the preflight to distinguish those cases so an absent `claude-code-hive` package can proceed as a first publish through the same tagged release workflow. Additionally, the preflight now reads the npm package name from each directory's `package.json` rather than assuming the directory name equals the npm name — making it resilient to scoped package names.

**Why it matters:** Operators should not need a hidden one-off bootstrap for the first Claude package release. If the workflow says it publishes a package, the first publish belongs inside the tested contract.

### 3 · Version-bearing release surfaces now move together again

This release also refreshes the root/workspace manifests, plugin manifests, exact packaged dependency pins, the hardcoded `@tctinh/agent-hive-mcp` runtime version string, tracked lockfiles, the changelog, and the PHILOSOPHY evolution log to `1.4.6`.

The Bun lock had drifted behind the current release line, so `v1.4.6` restores that parity instead of carrying the inconsistency forward.

**Why it matters:** Release checks only mean something when the docs, metadata, and packaged artifacts describe the same versioned contract.

## What Changed

### Added

- First-publish support in the npm publish preflight for `claude-code-hive`, so a package that does not exist yet can still publish through the normal tagged release flow
- The publish preflight now auto-derives the npm package name from each directory's `package.json`, supporting scoped package names cleanly

### Changed

- MCP runtime package renamed from `hive-mcp` to `@tctinh/agent-hive-mcp` (name was taken on npm)
- `claude-code-hive` dependency declaration updated to `@tctinh/agent-hive-mcp`
- Module specifiers in `launch-hive-mcp.mjs`, `prepare-workspace-hive-mcp.mjs`, and `verify-plugin-assets.mjs` updated to resolve `@tctinh/agent-hive-mcp`
- Root/workspace manifests, plugin manifests, packaged dependency pins, tracked lockfile markers, and release-facing docs now all report `1.4.6`
- Release docs now explain the first-publish path for `claude-code-hive` instead of assuming collaborator metadata already exists for a new package

### Fixed

- Release preparation no longer leaves the tracked Bun lock behind the current shipped version
- The first Claude package release no longer depends on an undocumented manual npm bootstrap outside the workflow contract
- The npm publish preflight no longer assumes directory name equals npm package name (fixes scoped packages)

---

## Breaking Changes

None.

This is a patch release focused on making the release contract for the new Claude package honest and testable, and unblocking the first publish by renaming the MCP runtime package to an available npm name.

## Philosophy Alignment

This release sharpens three of Hive's core principles:

**P2 (Plan → Approve → Execute):** The release path is more honest because the first publish of a new package is now part of the explicit contract instead of an implied manual exception.

**P7 (Iron Laws + Hard Gates):** The npm publish preflight remains strict for existing packages while learning the one missing case that the workflow now legitimately needs to handle. It also no longer assumes directory names equal npm package names, removing a fragile implicit coupling.

**P9 (Deterministic Contracts Beat Soft Memory):** Version-bearing metadata, operator docs, and release tests now move together again, reducing the chance of a "looks ready" release that still fails on an untested edge.

See the [PHILOSOPHY.md § v1.4.6](../../PHILOSOPHY.md) evolution note for the full rationale.
