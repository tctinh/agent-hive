# Release v1.4.6

Compare: [`v1.4.5...v1.4.6`](https://github.com/tctinh/agent-hive/compare/v1.4.5...v1.4.6)

---

## Highlights

### 1 · The first `claude-code-hive` publish is now part of the normal release path

`v1.4.6` closes the gap between the release workflow and the npm preflight for the first Claude package release.

The tagged release path already built, tested, packed, and ordered `claude-code-hive` behind `hive-mcp`, but the npm preflight still assumed every package already existed and already had collaborator metadata. That assumption was correct for existing packages and wrong for a brand-new one.

This patch teaches the preflight to distinguish those cases so an absent `claude-code-hive` package can proceed as a first publish through the same tagged release workflow.

**Why it matters:** Operators should not need a hidden one-off bootstrap for the first Claude package release. If the workflow says it publishes a package, the first publish belongs inside the tested contract.

### 2 · Version-bearing release surfaces now move together again

This release also refreshes the root/workspace manifests, plugin manifests, exact packaged dependency pins, the hardcoded `hive-mcp` runtime version string, tracked lockfiles, the changelog, and the PHILOSOPHY evolution log to `1.4.6`.

The Bun lock had drifted behind the current release line, so `v1.4.6` restores that parity instead of carrying the inconsistency forward.

**Why it matters:** Release checks only mean something when the docs, metadata, and packaged artifacts describe the same versioned contract.

## What Changed

### Added

- First-publish support in the npm publish preflight for `claude-code-hive`, so a package that does not exist yet can still publish through the normal tagged release flow

### Changed

- Root/workspace manifests, plugin manifests, packaged dependency pins, the `hive-mcp` runtime version string, tracked lockfile markers, and release-facing docs now all report `1.4.6`
- Release docs now explain the first-publish path for `claude-code-hive` instead of assuming collaborator metadata already exists for a new package

### Fixed

- Release preparation no longer leaves the tracked Bun lock behind the current shipped version
- The first Claude package release no longer depends on an undocumented manual npm bootstrap outside the workflow contract

## Philosophy Alignment

This release sharpens three of Hive's core principles:

**P2 (Plan → Approve → Execute):** The release path is more honest because the first publish of a new package is now part of the explicit contract instead of an implied manual exception.

**P7 (Iron Laws + Hard Gates):** The npm publish preflight remains strict for existing packages while learning the one missing case that the workflow now legitimately needs to handle.

**P9 (Deterministic Contracts Beat Soft Memory):** Version-bearing metadata, operator docs, and release tests now move together again, reducing the chance of a "looks ready" release that still fails on an untested edge.

See the [PHILOSOPHY.md § v1.4.6](../../PHILOSOPHY.md) evolution note for the full rationale.
